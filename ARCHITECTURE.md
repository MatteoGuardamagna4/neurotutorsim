# Phase 1 slice architecture

How the `be_001` vertical slice fits together: one unit, three instructional
conditions, fully validated, no TRIBE, no Colab. This document explains the
pipeline stage by stage, points at the file that implements each stage, and
gives a runnable example for each one. See `README.md` for setup/infra and
`CLAUDE.md` for project-wide decisions; this file is scoped to the Phase 1
slice only.

## The pipeline, end to end

```
data/interim/units/be_001.json   (authored source of truth)
        │
        ├─► scripts/build_units_csv.py ──► data/processed/units.csv (derived index)
        │
        ├─► src/validation/validators.py  (recompute + check every answer)
        │
stimuli/{traditional,ai_scaffolding,ai_substitution}/be_001_primary.txt
        │
        ├─► src/generation/stimulus_io.py   (strip front matter → body text)
        ├─► src/generation/features.py      (body → feature vector)
        ├─► src/generation/matching.py      (feature vectors → deltas/caliper)
        └─► src/validation/checks.py        (body → leakage / turn labels / readability)
        │
        ▼
scripts/run_slice.py ──► outputs/tables/slice_features.csv
                     └─► outputs/tables/slice_report.md
```

Nothing here touches TRIBE, Colab, or the learner engine — those are Phase II/III.

## 1. Authored content: `data/interim/units/be_001.json`

The **only** hand-authored file in the chain. Holds the canonical problem,
the scripted misconception, a practice problem, and near/far transfer
problems. Everything downstream is derived from it.

`data/processed/units.csv` is *never* hand-edited — it's regenerated:

```bash
python scripts/build_units_csv.py
```

This reads every `data/interim/units/*.json` and writes one CSV row per unit,
JSON-encoding the validator params with Python's `csv`/`json` modules so
quoting is always correct. Both files are git-tracked despite living under
`data/interim/` and `data/processed/`, which are otherwise gitignored — see
the `.gitignore` exceptions added for this pair.

## 2. Answer validation: `src/validation/validators.py`

Every problem in `be_001.json` names a validator by string. The validator
*recomputes* the answer from the problem's params — answers are never
hand-trusted.

```python
from src.validation import validators as V

V.get("break_even_quantity").check(
    params={"fixed_costs": 120000, "price": 80, "variable_cost": 50},
    answer=4000,
)
# ValidationResult(validator='break_even_quantity', expected=4000.0, passed=True, ...)

# the scripted misconception must NOT validate:
V.get("break_even_quantity").check(
    params={"fixed_costs": 120000, "price": 80, "variable_cost": 50},
    answer=1500,   # 120000 / 80, forgot to subtract variable cost
)
# passed=False
```

All five numeric validators raise `ValueError` on a non-positive contribution
margin (`price <= variable_cost`) or negative fixed costs — break-even is
undefined there, so the code refuses to return a number rather than silently
producing one.

## 3. Stimuli: `stimuli/<condition>/be_001_primary.txt`

Three files, same underlying problem, matched on length. Each starts with
`---`-delimited front matter (`stimulus_id`, `condition`, `example_count`,
`learner_error_scripted`, ...), stripped before the body is used anywhere
downstream:

```python
from src.generation.stimulus_io import parse_front_matter

meta, body = parse_front_matter("stimuli/ai_scaffolding/be_001_primary.txt")
meta["condition"]                 # "ai_scaffolding"
meta["learner_error_scripted"]    # True
```

`traditional` is static prose with a three-level hint ladder. `ai_scaffolding`
and `ai_substitution` are **hand-scripted** tutor/learner transcripts — the
learner's error (`120000 / 80 = 1500`) is fixed text, not generated by an
LLM or a real learner. That's a named limitation, not an oversight: real
learner-driven trajectories require the Phase III learner engine, which
doesn't exist yet.

## 4. Feature extraction: `src/generation/features.py`

Turns a body of text into the §5.3 observable-property vector: word/sentence
counts, Flesch–Kincaid grade, equation count, lexical diversity, and timing.

```python
from src.generation import features as F

fv = F.extract_features(body, example_count=meta["example_count"])
fv.words, fv.fk_grade, fv.duration_seconds
# (430, 2.77, 117.3)

F.word_onsets(body)[:3]   # [0.0, 0.27, 0.55]  -- seconds, at 220 wpm
```

`duration_seconds = 60 * words / 220` (reading at 220 words/minute — the same
number TTS rate would target in Phase II). `word_onsets` implements the same
formula per-word; it isn't consumed anywhere in this slice, but it's tested
and ready for when TRIBE needs word-level timing.

## 5. Matching diagnostics: `src/generation/matching.py`

Compares conditions against the `traditional` anchor.

```python
from src.generation import matching as M

M.raw_feature_deltas(anchor_features, scaffolding_features)
# {'words': -18, 'fk_grade': -2.84, 'duration_seconds': -4.91, ...}

M.duration_caliper_check(duration_anchor=122.2, duration_other=117.3)
# {'relative_difference': 0.040, 'tolerance': 0.10, 'passed': True}
```

`standardized_mean_difference` and `mahalanobis_distance` — the "real" §5.3
matching statistics — need a pooled variance/covariance estimated across a
*sample* of stimuli. With one unit there's no sample, so both raise
`ValueError` below 10 units rather than returning 0 or an approximation:

```python
M.standardized_mean_difference([430], [415])
# ValueError: standardized_mean_difference requires a pooled covariance/
# variance estimated across a sample of stimuli (§5.3), which needs >= 10
# units; got 1. This is unavailable at n=1 -- not zero, not an approximation.
```

## 6. Content validation: `src/validation/checks.py`

Three checks over a transcript body:

```python
from src.validation import checks as C

C.check_leakage(body, final_answer=4000)
# LeakageResult(leaked=False, first_attempt_turn_index=1, detail="...")

C.classify_tutor_turns(body)
# ['guiding_question', 'hint', 'guiding_question', 'hint', 'guiding_question', 'answer_reveal']

C.check_readability({"traditional": t_body, "ai_scaffolding": s_body})
# {'grades': {...}, 'spread': 2.84, 'flagged': True}
```

- `check_leakage` finds the learner's first numeric attempt, then checks
  whether the final answer already appeared in a tutor turn before it.
  Scaffolding must score `leaked=False`; substitution is *expected* to
  reveal early by design, so a `True` there is correct, not a bug.
- `classify_tutor_turns` labels each tutor turn `hint` (starts with the
  literal `Hint:` marker), `guiding_question` (contains `?`), or
  `answer_reveal` (neither — a declarative statement). This is a textual
  heuristic, not a semantic judgment, so the transcripts are written to be
  unambiguous under it.
- `check_readability` flags conditions whose Flesch–Kincaid grade spreads by
  more than 1.0 level. In this slice the scaffolding transcript reads several
  grades lower than the traditional prose (short back-and-forth turns vs.
  continuous exposition) — flagged, and reported, but not treated as a
  failure at n=1.

## 7. Running the whole slice: `scripts/run_slice.py`

```bash
python scripts/build_units_csv.py   # regenerate units.csv from the JSON
python scripts/run_slice.py         # run everything, write the report
```

Produces `outputs/tables/slice_features.csv` (one row per condition) and
`outputs/tables/slice_report.md` (validator results, feature table,
per-feature deltas, caliper check, the SMD/Mahalanobis n=1 note, leakage/
turn-classification results, and the scripted-transcript limitation). Both
files are regenerated from source on every run — never hand-edited, and
gitignored under `outputs/` since they're a build product, not source.

## The duration caliper, concretely

The caliper is the one §5.5 matching check that's meaningful at n=1: are the
three conditions roughly the same *length in time* to consume? If a
scaffolding tutor made you read for twice as long as the traditional prose,
any difference you later measured (plasticity, retention, whatever) could
just be a dose-of-exposure artifact, not an effect of the instructional
condition.

Concretely, `duration_caliper_check(duration_anchor, duration_other,
tolerance=0.10)` computes:

```
relative_difference = |duration_other - duration_anchor| / duration_anchor
passed = relative_difference <= tolerance
```

using `duration_seconds = 60 * words / 220` from each condition's feature
vector, with `traditional` as the anchor. In this slice: traditional is
~122.2s, ai_scaffolding ~117.3s (4.0% shorter, passes), ai_substitution
~118.4s (3.1% shorter, passes — this file was lengthened slightly during
authoring after an earlier draft came in at 10.3%, just over the 10% limit).
"Caliper" is borrowed from matching-methods terminology (as in propensity-
score caliper matching): a fixed tolerance band around the anchor value,
inside of which two stimuli count as acceptably matched on that one
dimension. It's deliberately a much weaker check than SMD/Mahalanobis — it
only checks one feature (duration) against a fixed threshold, rather than a
pooled multivariate distance — which is exactly why it still works with a
single unit.
